## 2. 月次記録画面の状態行列

### 2.1 月状態ごとの表示と操作

| ケース | 表示 | 編集 | `can_confirm` | `confirm_error_code` | UIメッセージの扱い |
| --- | --- | --- | --- | --- | --- |
| 対象月が確定済み | 閲覧専用レイアウト | 不可 | `false` | `FORBIDDEN` | `この月はすでに確定済みです。` |
| 対象月が先頭未確定月 | 入力レイアウト | 可 | 条件次第 | 条件次第 | 通常動作 |
| より古い未確定月が残る新しい月 | 入力レイアウト | 可 | `false` | `MONTH_NOT_CONFIRMABLE` | `先に古い未確定月を確定してください。` |
| 期限超過未確定月 | 入力レイアウト | 可 | 条件次第 | 条件次第 | 上部バナーのみ追加 |
| 資産 0 件 | 資産カードを空状態表示 | 資産追加のみ可 | `false` | `VALIDATION_ERROR` | `資産がありません。追加されるまで月次確定できません。` |
| 収入 0 件 | 収入カードを空状態表示 | 収入追加のみ可 | `false` | `VALIDATION_ERROR` | `収入がありません。追加されるまで月次確定できません。` |
| 資産と収入がともに 0 件 | 両カードを空状態表示 | 両方追加可 | `false` | `VALIDATION_ERROR` | `資産、収入がありません。追加されるまで月次確定できません。` |
| 投資先 0 件 | 投資カードを説明のみ空状態表示 | 設定画面への導線のみ | 他入力が揃えば `true` | `null` | 月次確定は可能 |
| 空欄あり | 入力レイアウト | 可 | `false` | `VALIDATION_ERROR` | `未入力の項目があります。` |
| 入力完了 | 入力レイアウト | 可 | `true` | `null` | 確定バーを有効化 |

### 2.2 `GET /monthly-records/{month}` の `confirm_error_details.categories`

| 状態 | categories |
| --- | --- |
| 資産 0 件 | `["assets"]` |
| 収入 0 件 | `["incomes"]` |
| 資産・収入 0 件 | `["assets", "incomes"]` |
| 資産に空欄 | `["assets"]` |
| 収入に空欄 | `["incomes"]` |
| 投資評価額に空欄 | `["investment_valuations"]` |
| 複数区分に空欄 | 該当区分をすべて列挙 |

- `categories` は affected category の列挙に留める。
- `0件` と `空欄` の文言差分は backend が `confirm_error` に埋める。
- frontend は `confirm_error_code` と `confirm_error` をそのまま表示し、独自再判定をしない。

## 3. 更新系 API の扱い

### 3.1 月次更新 API 共通

- 月次系の更新 API はすべて即保存で扱う。
- 確定済み月に対する更新系 API はすべて `403 FORBIDDEN` とする。
- 値未入力は「値行なし」または `null` で表現し、ダミー値を保存しない。
- `POST /monthly-records/{month}/confirm` では frontend が常に「最終的に採用する支出値」を `expense_override_yen` として送る。

### 3.2 資産操作

| 操作 | 可能条件 | 反映開始 | 具体ルール |
| --- | --- | --- | --- |
| 資産追加 | 対象月が未確定 | 選択月 | `asset_definition` 作成、`asset_name_revision(effective_month=選択月)` 作成、表示期間は選択月から開始 |
| 資産名編集 | 対象月が未確定 | 選択月 | 過去確定月の名称は変更しない。選択月以降にだけ新名称を適用する |
| 当月値更新 | 対象月が未確定 | 選択月のみ | `monthly_asset_values` を upsert する |
| 資産削除 | 任意の未確定月 | `first_unconfirmed_month` | 選択月に関係なく、最新確定月の次の未確定月から非表示にする |
| 資産復活 | 任意の未確定月 | `first_unconfirmed_month` | 直近の削除期間を閉じ、最新確定月の次の未確定月から再表示する |

- 資産削除と復活は「閲覧中の月」ではなく「最新確定月境界」で効く。
- `effective_from_month` は常に backend が返し、frontend はそれを表示に使う。

### 3.3 収入操作

| 操作 | 可能条件 | 反映開始 | 具体ルール |
| --- | --- | --- | --- |
| 収入追加 | 対象月が未確定 | 選択月 | `income_definition` と表示期間を選択月起点で作成する |
| 収入名編集 | 対象月が未確定 | 選択月 | 選択月以降にだけ新名称を適用する |
| 当月値更新 | 対象月が未確定 | 選択月のみ | `monthly_income_values` を upsert する |
| 収入削除 | 対象月が未確定 | 選択月 | `effective_from_month = 選択月` として以後非表示にする |

- 収入には restore API を持たない。
- 収入削除は資産削除と違い、最新確定月境界ではなく「選択月」起点で適用する。

### 3.4 投資評価額操作

| 操作 | 可能条件 | 反映開始 | 具体ルール |
| --- | --- | --- | --- |
| 投資評価額更新 | 対象月が未確定かつ投資先定義がある | 選択月のみ | `monthly_investment_valuations` を upsert する |

- 投資先 0 件のときは `investment_valuations: []` を返し、更新 API を呼ばない。
- `value_yen` は `0` 以上とし、負数は受け付けない。

## 4. 月次確定

### 4.1 `POST /monthly-records/{month}/confirm`

- 先頭未確定月だけが確定対象になる。
- 対象月に空欄がある場合は `409 VALIDATION_ERROR`。
- 資産 0 件または収入 0 件でも `409 VALIDATION_ERROR`。
- 既に確定済みなら `403 FORBIDDEN`。
- 古い未確定月が残っていれば `409 MONTH_NOT_CONFIRMABLE`。
- 成功時は以下を同一トランザクションで行う。
  - `confirmed=true`
  - `confirmed_at=now`
  - `expense_final_yen=expense_override_yen`
  - 必要なら次月 `monthly_record` を 1 件だけ追加

### 4.2 次月自動追加の idempotency

- 自動追加は「確定した月が管理中最後の月だったとき」だけ行う。
- 追加対象は `confirmed month + 1 month` の 1 件だけ。
- すでにその月の `monthly_record` が存在する場合は追加しない。
- 同じ確定 API の再送で重複レコードを作らない。

## 5. 設定改定の扱い

### 5.1 `effective_from_month`

- 設定系 API は毎回 `first_unconfirmed_month` を `effective_from_month` として使う。
- 選択中の画面月に引きずられない。
- ダッシュボードの分析月が `2026-04` でも、`GET /settings` は `2026-05` 反映予定の設定を返してよい。

### 5.2 2 API の同月マージ

| API | 既存 revision が同月にある場合 | 無い場合 |
| --- | --- | --- |
| `PATCH /settings/emergency-fund` | `emergency_fund_yen` だけ更新し、配分は維持 | 新規 revision 作成 |
| `PUT /settings/investment-target-allocations` | 配分を全置換し、`emergency_fund_yen` は維持 | 新規 revision 作成 |

- 同じ `user_id + effective_month` に対して revision は 1 行だけとする。
- 配分の保存は毎回全置換とし、部分更新しない。
