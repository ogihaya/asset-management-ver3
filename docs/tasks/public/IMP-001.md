# IMP-001: 公開 API 仕様と現行 auth PoC の差分整理および置換方針確定

## 🎯 概要

公開 auth API 仕様と現行 backend/frontend の auth PoC の差分を棚卸しし、後続の IMP-002、IMP-003、IMP-004 が迷わず進められる置換方針を確定する。  
このタスクではコード実装は行わず、差分整理、影響範囲、採用方針、引き継ぎ条件を文書で固定する。  
正式な成果物は `docs/requirements/public/AUTH_IMPLEMENTATION_DETAIL.md` とする。

## 📝 詳細説明

### 背景

現行実装は `login_id + password`、`access_token` Cookie、`is_authenticated` 前提の暫定 PoC であり、公開仕様の `email + password + asset_management_session + data/authenticated` とは契約が異なる。  
この状態のまま IMP-002 以降を進めると、backend と frontend の両方で暫定仕様を増築してしまい、後戻りコストが上がる。

IMP-001 の役割は、以下を明確にすることにある。

- 何が正式仕様と食い違っているか
- どこを置換対象にするか
- 後続タスクが何を前提に実装すべきか
- どのファイル群が変更影響を受けるか

### 完了条件

IMP-001 は、以下を満たした時点で完了とする。

1. backend の現行 auth 契約差分が整理されていること
2. frontend の現行 auth 契約差分が整理されていること
3. 差分ごとに「正式仕様」「現状」「採用方針」が 1 つの表で確認できること
4. IMP-002、IMP-003、IMP-004 が依存する前提条件が明文化されていること
5. 「今回はまだコードを触らない」ことが明記されていること
6. 上記の内容が `docs/requirements/public/AUTH_IMPLEMENTATION_DETAIL.md` に反映されていること

### 実装内容

#### Step 1: backend の現行 auth 契約を棚卸しする

- login request が `login_id` を受け取っている箇所を洗い出す
- login response が `access_token` を返している箇所を洗い出す
- Cookie 名が `access_token` になっている箇所を洗い出す
- PoC の仮実装である点を明示する
- `User` と `UserRepository` が `email` ではなく `login_id` 前提になっている点を整理する

#### Step 2: frontend の現行 auth 契約を棚卸しする

- auth types が `login_id`、`access_token`、`is_authenticated` 前提になっている箇所を洗い出す
- middleware が `access_token` Cookie 前提で動いている点を整理する
- 保護対象 route が `/dashboard` のみである点を整理する
- login 以外の public auth 画面が未整備である点を整理する

#### Step 3: 差分表を作成する

少なくとも以下の論点は差分表に含める。

- ログイン識別子
- 認証 Cookie 名
- 認証方式
- `/auth/status` のレスポンス形状
- signup の有無
- password reset の有無
- 保護対象 route
- signup 後の遷移
- frontend auth store が保持すべき値

差分表は以下の 3 列を最低限持つ。

| 論点 | 現行実装 | 正式仕様 / 採用方針 |
| --- | --- | --- |

#### Step 4: 置換方針を固定する

各差分に対して、以下を判断して明記する。

- 維持しない旧契約
- 後続タスクで置換する契約
- 置換対象レイヤ
- 置換順序

本タスクで固定すべき採用方針は以下とする。

- `login_id` は維持せず `email` に置き換える
- `access_token` Cookie は維持せず `asset_management_session` に置き換える
- `is_authenticated` 前提は維持せず `data.authenticated` に合わせる
- auth guard の保護対象は `/dashboard` だけで終わらせず、`/monthly-record` `/life-plan` `/settings` まで広げる
- signup 成功後に自動ログインはしない

#### Step 5: 後続タスクへの引き継ぎ条件を整理する

IMP-002、IMP-003、IMP-004 に対して、最低限以下を引き継ぐ。

- IMP-002:
  - `users / user_sessions / password_reset_tokens` を前提に DB モデル設計を行う
  - `login_id` 前提のテーブルや型を増やさない
- IMP-003:
  - Cookie session を前提に backend auth を作り直す
  - `access_token` 発行型の設計を増築しない
- IMP-004:
  - frontend の auth types、API client、middleware を正式 API に合わせて作り直す
  - 旧型互換レイヤは原則作らない

### 非対象

以下は IMP-001 の対象外とする。

- DB migration の実装
- session 実装そのもの
- signup/login/logout/status API のコード更新
- password reset 実装
- frontend 画面実装
- テストコードの修正

## 💻 コード例

### 差分整理で固定すべき API 例

#### 現行 PoC

```json
{
  "login_id": "admin",
  "password": "pass"
}
```

```json
{
  "access_token": "token",
  "user_id": 1
}
```

#### 正式仕様

```json
{
  "email": "user@example.com",
  "password": "Passw0rd"
}
```

```json
{
  "data": {
    "user": {
      "id": 1,
      "email": "user@example.com"
    }
  },
  "meta": {
    "session_expires_in_days": 30
  }
}
```

### 差分整理で固定すべき Cookie 例

```text
before: access_token=...
after:  asset_management_session=...
```
