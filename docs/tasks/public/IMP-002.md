## 🎯 概要

`users`、`user_sessions`、`password_reset_tokens` の 3 テーブルについて、公開 DB 設計どおりの SQLAlchemy モデルと Alembic 初回 migration を実装する。  
このタスクでは、既存 `User` 周辺の `login_id` 前提を新スキーマへ最小同期するところまで含める。  
正式な正本は `docs/requirements/public/DATABASE_DESIGN.md`、`docs/requirements/public/database.dbml`、`docs/requirements/public/AUTH_IMPLEMENTATION_DETAIL.md` とする。

## 📝 詳細説明

### 背景

現状の backend には auth PoC 用の `users` モデルしかなく、公開設計との不整合が残っている。

- `backend/app/infrastructure/db/models/user_model.py` は `login_id` と `password` を持つ暫定定義である
- `backend/app/domain/entities/user.py` と `backend/app/domain/repositories/user_repository.py` も `login_id` 前提である
- `backend/alembic/env.py` は `target_metadata = None` のままで、autogenerate に必要な model 集約が未整備である
- `backend/alembic/versions/` には migration ファイルがまだ存在しない

この状態では IMP-003 以降の session 認証実装が始められないため、まず DB 基盤を公開設計どおりに揃える必要がある。

### 完了条件

IMP-002 は、以下を満たした時点で完了とする。

1. `users`、`user_sessions`、`password_reset_tokens` の SQLAlchemy モデルが公開 DB 設計どおりに定義されていること
2. Alembic が `Base.metadata` を参照できる状態になっていること
3. auth 3 テーブルのみを対象とする初回 migration が生成・調整されていること
4. `User` entity、`IUserRepository`、`UserRepositoryImpl` が新スキーマへ最小同期されていること
5. 既存の user 系テストが新スキーマ前提へ更新されていること
6. 認証 API の挙動変更や session 利用ロジック追加は本タスクに含めないことが明記されていること

### 実装内容

#### Step 1: auth 3 テーブルの列定義を正本に合わせて固定する

以下を `DATABASE_DESIGN.md` と `database.dbml` から確定し、実装前提とする。

- `users`
  - `email`, `password_hash`, `name`, `deleted_at`, `created_at`, `updated_at`
- `user_sessions`
  - `user_id`, `session_token_hash`, `expires_at`, `revoked_at`, `last_seen_at`, `created_at`
- `password_reset_tokens`
  - `user_id`, `token_hash`, `expires_at`, `used_at`, `created_at`

ここで固定する制約は次のとおり。

- `users.email` は unique
- `user_sessions.session_token_hash` は unique
- `password_reset_tokens.token_hash` は unique
- `user_sessions.user_id` と `password_reset_tokens.user_id` は `users.id` への FK を持つ
- `users` は論理削除前提で `deleted_at` を持つ

#### Step 2: SQLAlchemy モデルを実装する

- `user_model.py` を `login_id/password` 前提から `email/password_hash/deleted_at/created_at/updated_at` 前提へ置き換える
- `user_session_model.py` を追加する
- `password_reset_token_model.py` を追加する
- index、unique、nullable、default は公開設計どおりに定義する
- FK 削除方針は `CASCADE` ではなく `RESTRICT` 前提で進める

#### Step 3: model 集約と Alembic metadata を有効化する

- `backend/app/infrastructure/db/models/__init__.py` を作成し、auth 3 モデルを export する
- 必要であれば `backend/app/infrastructure/db/__init__.py` も整備し、Alembic から `Base` と model import ができる形にする
- `backend/alembic/env.py` で `target_metadata = Base.metadata` を有効化する
- Alembic の autogenerate が auth 3 テーブルを検出できる状態にする

#### Step 4: 初回 migration を作成・調整する

- `make db-migrate msg='create auth base tables'` で初回 migration を生成する
- 生成後の migration は必ず目視確認し、必要なら手修正する
- migration の対象は auth 3 テーブルのみに限定する
- 確認対象は次のとおり
  - テーブル名
  - カラム型
  - nullable
  - default
  - unique
  - index
  - FK
  - downgrade の整合

#### Step 5: 既存 User 周辺を新スキーマへ最小同期する

IMP-002 では互換レイヤを作らず、既存 `User` 周辺を新スキーマへ最小同期する。

- `User` entity は `login_id` と `password` をやめ、`email` と `password_hash` を正とする
- `deleted_at` を entity で保持できるようにする
- `IUserRepository.get_by_login_id` は `get_by_email` に置き換える
- `UserRepositoryImpl` は `UserModel.email` と `UserModel.password_hash` を使うように更新する
- `delete` は物理削除ではなく `deleted_at` 更新に変更する

この同期は「DB モデルと repository が壊れないようにするため」の最小範囲に留める。  
認証 API や use case の挙動変更は IMP-003 に残す。

#### Step 6: 既存 user 系テストを新スキーマ前提に更新する

最低限、以下の観点で既存テストを更新する。

- `User` entity が `email` `password_hash` `deleted_at` 前提で生成できる
- `IUserRepository` が `get_by_email` を持つ
- `UserRepositoryImpl` が email で取得できる
- `delete` が論理削除として振る舞う

必要なら `backend/tests/conftest.py` の DB 初期化前提も調整する。

### 実装時の判断ルール

本タスクでは次の判断を固定する。

- `users` は論理削除前提とし、物理削除はしない
- `login_id` は復活させない
- `password` 平文カラムは残さず `password_hash` を正とする
- session と reset token の repository interface / 実装追加は IMP-003 以降に回す
- auth API の request/response 変更は行わない
- auth 3 テーブル以外の migration は作らない

### 非対象

以下は IMP-002 の対象外とする。

- `POST /auth/login` `POST /auth/logout` `GET /auth/status` の実装変更
- signup、password reset、`GET /users/me`、`DELETE /users/me` の実装
- `security_service_impl.py` の session 化
- `UserSessionRepository`、`PasswordResetTokenRepository` の interface / 実装追加
- asset、income、investment target、settings、monthly records、life plan の DB モデルや migration

## 💻 コード例

### `users` の before / after

#### 現状

```python
class UserModel(Base):
    __tablename__ = 'users'

    id = Column(Integer, primary_key=True, autoincrement=True)
    login_id = Column(String(255), unique=True, nullable=False, index=True)
    password = Column(String(255), nullable=False)
    email = Column(String(255), nullable=True)
    name = Column(String(255), nullable=True)
```

#### IMP-002 完了後の方向

```python
class UserModel(Base):
    __tablename__ = 'users'

    id = Column(Integer, primary_key=True, autoincrement=True)
    email = Column(String(255), unique=True, nullable=False, index=True)
    password_hash = Column(String(255), nullable=False)
    name = Column(String(255), nullable=True)
    deleted_at = Column(DateTime(timezone=True), nullable=True)
    created_at = Column(DateTime(timezone=True), nullable=False, server_default=func.now())
    updated_at = Column(DateTime(timezone=True), nullable=False, server_default=func.now(), onupdate=func.now())
```

## 📝 備考

- 本チケットは docs 整理ではなく、backend の DB 基盤実装タスクである。
