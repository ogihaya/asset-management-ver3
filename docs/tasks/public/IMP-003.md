## 🎯 概要

メールアドレスとパスワードを用いた Cookie session 認証へ移行するために、backend の lower-level 基盤を実装する。  
このタスクでは `UserSession` の domain / repository / security service / DI を整備し、後続の IMP-101 と IMP-102 が正式 API を迷わず実装できる状態を作る。  
正式な正本は `docs/requirements/public/IMPLEMENTATION_PLAN.md`、`docs/requirements/public/API_SPECIFICATION.md`、`docs/requirements/public/DATABASE_DESIGN.md`、`docs/requirements/public/AUTH_IMPLEMENTATION_DETAIL.md` とする。

IMP-003 は **session 基盤のみ** を対象とし、`signup`、`login`、`logout`、`auth status`、`users/me`、`delete me` の正式 API 実装は含めない。  
また、`password_reset_tokens` の domain / repository 基盤も本タスクでは作らず、IMP-102 に残す。

## 📝 詳細説明

### 背景

IMP-002 により `users`、`user_sessions`、`password_reset_tokens` の DB モデルと初回 migration は整備された。  
一方で現行 backend の認証実装は、依然として以下の PoC 状態にある。

- `backend/app/presentation/api/auth_api.py` は `login_id` と `access_token` Cookie を前提にしている
- `backend/app/application/use_cases/auth_usecase.py` は `admin/pass` をハードコードしている
- `backend/app/infrastructure/security/security_service_impl.py` は JWT を Cookie に入れる構成であり、DB 永続化された session を使っていない
- `backend/app/domain/` には `UserSession` entity や repository interface がまだ存在しない
- `backend/app/application/interfaces/security_service.py` も JWT token 発行中心の PoC 形になっている

この状態では、IMP-101 で正式な `signup/login/logout/status/users/me/delete me` を実装しようとしても、session 永続化・token hash・password hash・session 検証の責務が未整理のままである。  
IMP-003 の役割は、これらの lower-level 基盤を先に整備し、PoC API を増築せず正式仕様へ置換できる前提を作ることにある。

### 完了条件

IMP-003 は、以下を満たした時点で完了とする。

1. `UserSession` entity と `IUserSessionRepository` が追加されていること
2. `UserSessionRepositoryImpl` が `user_sessions` を扱えること
3. `IPasswordHashService` と `ISessionTokenService` が追加され、infra 実装があること
4. session token の平文生成、hash 化、session 検証、1日1回までの延長判定に必要な lower-level 責務が定義されていること
5. 後続 use case が `UserRepository`、`IUserSessionRepository`、`IPasswordHashService`、`ISessionTokenService` を DI で受け取れる前提が整っていること
6. 旧 PoC の `auth_api.py`、`AuthUsecase`、JWT 実装は本タスクではまだ置き換えないことが明記されていること
7. Alembic migration や public API 契約変更を本タスクに含めないことが明記されていること

### 実装内容

#### Step 1: IMP-003 の対象外を最初に固定する

このタスクでは、以下を **やらない** と最初に固定する。

- `auth_api.py` の正式置換
- `POST /auth/login`、`POST /auth/logout`、`GET /auth/status` の request / response 正式化
- `POST /auth/signup`
- `GET /users/me`
- `DELETE /users/me`
- `POST /auth/password-reset/request`
- `POST /auth/password-reset/reset`
- `password_reset_tokens` の domain / repository 追加
- 新しい Alembic migration 作成

IMP-003 は「後続 API 実装のための session 基盤づくり」に限定する。

#### Step 2: `UserSession` entity と repository interface を追加する

`backend/app/domain/` に session 永続化を表す最小単位を追加する。

- `UserSession` entity は少なくとも以下を持つ前提で設計する
  - `id`
  - `user_id`
  - `session_token_hash`
  - `expires_at`
  - `revoked_at`
  - `last_seen_at`
  - `created_at`
- entity では、少なくとも以下の判断が読める形にする
  - 有効 session か
  - 失効済みか
  - 期限切れか
  - `last_seen_at` に基づいて延長対象か

`IUserSessionRepository` は、後続 use case が必要とする最小責務に絞る。

- `create`
- `find_active_by_token_hash`
- `revoke_by_token_hash`
- `revoke_all_by_user_id`
- `touch`

#### Step 3: security 系 interface を分離する

現行 `ISecurityService` は JWT access token 前提の PoC であり、IMP-003 の正本にはならない。  
ただし IMP-003 中は壊さず、一時共存を許可する。

本タスクでは、新たに以下を追加する。

- `IPasswordHashService`
  - `hash_password(plain_password)`
  - `verify_password(plain_password, password_hash)`
- `ISessionTokenService`
  - `generate_token()`
  - `hash_token(raw_token)`

判断ルールは次のとおり。

- password hash は既存の passlib / bcrypt を流用する
- session token は十分なランダム性を持つ平文 token を生成する
- DB に保存するのは `session_token_hash` のみとし、raw token は保存しない

#### Step 4: infrastructure を実装する

`backend/app/infrastructure/` に lower-level 実装を追加する。

- `UserSessionRepositoryImpl`
  - `user_sessions` テーブルを扱う
  - token hash で active session を引ける
  - current session だけの失効と、user 単位の全失効ができる
  - 1 日以上経過時だけ `last_seen_at` / `expires_at` を更新できる前提を持つ
- `PasswordHashServiceImpl`
  - `hash_password`
  - `verify_password`
- `SessionTokenServiceImpl`
  - raw token 生成
  - token hash 生成

#### Step 5: session 検証基盤の lower-level 前提を整える

後続の `login/logout/status/users/me/delete me` が使えるよう、以下の前提をコード上で表現できる状態にする。

- Cookie 名は `asset_management_session`
- session 有効期限は 30 日
- 認証済み判定には以下を使う
  - Cookie が存在する
  - token hash が一致する
  - `revoked_at is null`
  - `expires_at > now`
  - 対象 `users.deleted_at is null`
- `last_seen_at is null` または `last_seen_at <= now - 1 day` のときだけ延長対象

この段階では presentation / use case に完全接続しなくてもよいが、後続タスクが new service / repository をそのまま使える粒度まで実装する。

#### Step 6: DI を整える

後続タスクが基盤をすぐ使えるように、DI で次を解決できる状態を作る。

- `UserRepository`
- `IUserSessionRepository`
- `IPasswordHashService`
- `ISessionTokenService`

ただし、現行 PoC の `get_auth_usecase()` は IMP-003 中は壊さない。  
IMP-101 で正式 auth use case へ置換する前提で、一時共存を許可する。

#### Step 7: Docker 前提のテストを追加する

Docker 起動済み前提で、backend コンテナ内から検証できるようにする。

少なくとも以下の新規テストを用意する前提で進める。

- `tests/domain/test_user_session_entity.py`
- `tests/domain/test_user_session_repository_interface.py`
- `tests/infrastructure/test_user_session_repository_impl.py`
- `tests/infrastructure/test_password_hash_service_impl.py`
- `tests/infrastructure/test_session_token_service_impl.py`

回帰確認として、以下も流す。

- `tests/application/test_auth_usecase.py`
- `tests/presentation/test_auth_api.py`
- `tests/infrastructure/test_user_repository_impl.py`

### 実装時の判断ルール

本タスクでは、以下の判断を固定する。

- IMP-003 は session 基盤のみを対象とする
- `password_reset_tokens` の domain / repository は IMP-102 に残す
- public API の request / response は変えない
- Cookie 名の実運用切り替えも IMP-101 に残す
- raw session token は DB 保存しない
- session 有効期限は 30 日を前提にする
- 複数端末ログインを許可する
- session 延長は最大 1 日 1 回とする
- 旧 `ISecurityService` と `SecurityServiceImpl` は IMP-003 中は一時共存を許可する
- 旧 PoC の auth API は壊さない

### 非対象

以下は IMP-003 の対象外とする。

- `POST /auth/login` の正式レスポンス化
- `POST /auth/logout` の正式レスポンス化
- `GET /auth/status` の `data.authenticated` 形への置換
- `POST /auth/signup`
- `GET /users/me`
- `DELETE /users/me`
- `POST /auth/password-reset/request`
- `POST /auth/password-reset/reset`
- `PasswordResetToken` entity / repository
- frontend の auth 型、middleware、画面更新
- Alembic migration の追加

## 💻 コード例

### IMP-003 で追加する interface の方向

```python
class IUserSessionRepository(ABC):
    @abstractmethod
    def create(self, session: UserSession) -> UserSession:
        pass

    @abstractmethod
    def find_active_by_token_hash(self, session_token_hash: str) -> UserSession | None:
        pass

    @abstractmethod
    def revoke_by_token_hash(self, session_token_hash: str, revoked_at: datetime) -> bool:
        pass
```

```python
class IPasswordHashService(ABC):
    @abstractmethod
    def hash_password(self, plain_password: str) -> str:
        pass

    @abstractmethod
    def verify_password(self, plain_password: str, password_hash: str) -> bool:
        pass
```

```python
class ISessionTokenService(ABC):
    @abstractmethod
    def generate_token(self) -> str:
        pass

    @abstractmethod
    def hash_token(self, raw_token: str) -> str:
        pass
```
