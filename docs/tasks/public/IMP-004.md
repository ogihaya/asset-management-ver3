# IMP-004: frontend の auth 基盤を公開仕様へ再構成

## 🎯 概要

frontend の auth 基盤を、現行の `login_id` / `access_token` / `is_authenticated` 前提から、公開仕様の `email` / `asset_management_session` / `data.authenticated` 前提へ再構成する。  
このタスクでは **API client、型、auth state、auth bootstrap、middleware、route guard、テスト方針** を整備し、後続の IMP-103 が auth 画面を迷わず実装できる状態を作る。  
正式な正本は `docs/requirements/public/IMPLEMENTATION_PLAN.md`、`docs/requirements/public/AUTH_IMPLEMENTATION_DETAIL.md`、`docs/requirements/public/FRONTEND_MIGRATION_MAP.md`、`docs/requirements/public/API_SPECIFICATION.md` とする。

IMP-004 は **frontend 基盤のみ** を対象とし、`/login` `/signup` `/password-reset/request` `/password-reset/reset` の画面本体実装は含めない。  
また、backend 側の auth API 実装変更や、PoC 互換層の追加も本タスクでは行わない。

## 📝 詳細説明

### 背景

現行 frontend の auth 実装は、公開仕様ではなく auth PoC を前提にしている。

- `frontend/src/entities/auth/model/types.ts` は `login_id`、`access_token`、`is_authenticated` を前提にしている
- `frontend/src/entities/auth/api/auth-api.ts` は PoC の request / response shape をそのまま使っている
- `frontend/src/middleware.ts` は `access_token` Cookie を見て `/dashboard` だけを保護している
- `frontend/src/app/providers.tsx` は PoC `/auth/status` 契約で auth bootstrap をしている
- `frontend/src/__tests__/middleware.test.ts` と `frontend/src/entities/auth/__tests__/auth-api.test.ts` も旧契約に依存している
- `/signup` `/password-reset/request` `/password-reset/reset` の route はまだ存在しない

この状態で IMP-103 の画面実装へ進むと、画面側で旧契約依存が広がり、後からの置換コストが増える。  
IMP-004 の役割は、先に frontend の auth foundation を正式仕様前提で整備し、画面実装を PoC 互換ではなく正式仕様上に載せることにある。

### 完了条件

IMP-004 は、以下を満たした時点で完了とする。

1. `entities/auth` の型定義が公開仕様前提に更新されていること
2. `entities/auth/api/auth-api.ts` が正式 auth API 契約前提で再構成されていること
3. Redux の auth state が `user`、`isAuthenticated`、`isLoading` を正式仕様前提で扱えること
4. `app/providers.tsx` の auth bootstrap が正式 `/auth/status` 応答前提で動くこと
5. `middleware.ts` が `asset_management_session` と正式 route 群前提で動くこと
6. `SESSION_EXPIRED` を frontend 側で処理する導線が定義されていること
7. auth 画面本体実装と backend 実装変更を本タスクに含めないことが明記されていること
8. 旧 `login_id` / `access_token` / `is_authenticated` の互換層を作らないことが明記されていること

### 実装内容

#### Step 1: IMP-004 の対象外を最初に固定する

このタスクでは、以下を **やらない** と最初に固定する。

- `/login` 画面の本格的な UI 改修
- `/signup` `/password-reset/request` `/password-reset/reset` の画面本体実装
- dashboard / monthly-record / life-plan / settings の本実装
- backend の auth API 実装変更
- PoC と正式仕様の二重対応互換層

IMP-004 は「frontend の auth foundation を正式仕様へ置き換える」ことに限定する。

#### Step 2: `entities/auth` の型を正式仕様へ置換する

`frontend/src/entities/auth/model/types.ts` を公開仕様へ合わせる。

- `LoginRequest` は `email` と `password` を持つ
- `LoginResponse` は `data.user` と `meta.session_expires_in_days` を持つ
- `StatusResponse` は `data.authenticated` と `data.user` を持つ
- `LogoutResponse` は正式仕様 shape に揃える
- `User` 型は `id`、`email`、`name` を正とする

判断ルールは次のとおり。

- `access_token` は response body 型に持たない
- `login_id` は型に残さない
- `is_authenticated` は top-level に残さない

#### Step 3: `entities/auth/api/auth-api.ts` を正式契約前提で再構成する

`frontend/src/entities/auth/api/auth-api.ts` の責務を次で固定する。

- `login`
- `logout`
- `getAuthStatus`

必要であれば、後続タスクが使えるように以下の関数シグネチャだけ先に用意してよい。

- `signup`
- `passwordResetRequest`
- `passwordResetReset`

ただし IMP-004 時点では UI から未使用でもよく、backend 未実装 API の実疎通を完了条件には含めない。

#### Step 4: auth state を正式仕様前提で整える

`frontend/src/store/slices/authSlice.ts` と関連 hook を正式仕様前提で再構成する。

- auth state は `user | null`、`isAuthenticated`、`isLoading` のみを持つ
- `access_token` や旧型の token 状態は持たない
- `user` は `entities/auth` の正式 `User` 型へ揃える
- `isLoading` は auth bootstrap の完了待ちを表す

#### Step 5: auth bootstrap を `/auth/status` 正式応答前提へ更新する

`frontend/src/app/providers.tsx` の auth bootstrap を公開仕様へ揃える。

- `GET /auth/status` の `data.authenticated` を正とする
- 認証済みなら `user` を hydrate する
- 未認証なら clear する
- `401 SESSION_EXPIRED` を受けたら clear して login へ戻す導線を作る

この時点では toast 表示まで必須ではないが、後続で入れられる構造にはしておく。

#### Step 6: middleware を正式仕様へ置換する

`frontend/src/middleware.ts` の責務を次で固定する。

- Cookie 名は `asset_management_session`
- protected path は `/dashboard` `/monthly-record` `/life-plan` `/settings`
- public auth path は `/login` `/signup` `/password-reset/request` `/password-reset/reset`
- middleware は `/auth/status` を使って認証を確認する
- middleware は Cookie の中身を decode しない
- `/auth/status` に失敗した場合は未認証扱いに倒す

#### Step 7: テストと mock を正式仕様へ更新する

少なくとも以下の更新を前提に進める。

- `frontend/src/entities/auth/__tests__/auth-api.test.ts`
- `frontend/src/__tests__/middleware.test.ts`
- `frontend/src/__mocks__/handlers/auth.ts`

確認観点は次のとおり。

- login request が `email` を送る
- auth status が `data.authenticated` を正として扱う
- `asset_management_session` 前提で route guard が動く
- `SESSION_EXPIRED` を未認証扱いへ落とせる

### 実装時の判断ルール

本タスクでは、以下の判断を固定する。

- IMP-004 は frontend の auth foundation だけを対象とする
- auth API 契約は **公開仕様を正** とする
- backend がまだ未実装の API については IMP-101 / IMP-102 待ちを許容する
- `login_id`、`access_token`、`is_authenticated` の互換層は作らない
- protected route は `/dashboard` `/monthly-record` `/life-plan` `/settings`
- public auth route は `/login` `/signup` `/password-reset/request` `/password-reset/reset`
- middleware と provider は cookie の decode をしない
- auth state は token を保持しない

### backend 依存の扱い

IMP-004 は frontend 側の foundation 整備を先行して進める。  
ただし backend の正式 auth API 実装は IMP-101 / IMP-102 に残るため、以下を前提にする。

- 型と API client は正式仕様で固定してよい
- 実疎通が未成立でもチケット上は問題としない
- 画面本体実装は backend 側の正式 API 実装後に IMP-103 で接続する

### 非対象

以下は IMP-004 の対象外とする。

- `/login` の本格 UI 改修
- `/signup` `/password-reset/request` `/password-reset/reset` の画面実装
- dashboard / monthly-record / life-plan / settings の本実装
- backend auth API の実装変更
- PoC と正式仕様の二重対応
