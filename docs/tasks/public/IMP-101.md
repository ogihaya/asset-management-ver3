# IMP-101: signup / login / logout / auth status / users me を正式仕様で backend 実装

## 🎯 概要

backend の auth PoC を、公開仕様の Cookie session 認証 API へ置換する。  
このタスクでは `POST /api/v1/auth/signup` `POST /api/v1/auth/login` `POST /api/v1/auth/logout` `GET /api/v1/auth/status` `GET /api/v1/users/me` `DELETE /api/v1/users/me` を正式実装し、frontend が前提としている runtime 契約と一致させる。  
正式な正本は `docs/requirements/public/IMPLEMENTATION_PLAN.md`、`docs/requirements/public/API_SPECIFICATION.md`、`docs/requirements/public/REQUIREMENTS.md`、`docs/requirements/public/AUTH_IMPLEMENTATION_DETAIL.md` とする。

IMP-101 は **backend の auth / account API 正式化** のみを対象とし、password reset は IMP-102、auth 画面本体は IMP-103、frontend の追加調整は IMP-104 に残す。  
また、PoC 互換 endpoint や `login_id` / `access_token` / JWT cookie の runtime 互換層は本タスクでは作らない。

## 📝 詳細説明

### 背景

IMP-003 までで、backend には正式 auth 実装に必要な session 基盤が揃っている。

- `users` と `user_sessions` は email / password_hash / logical delete / session hash 前提になっている
- `UserRepository`、`UserSessionRepository`、`PasswordHashService`、`SessionTokenService` は利用可能である
- frontend はすでに `email`、`asset_management_session`、`data.authenticated`、`/api/v1` 前提へ移行済みである

一方、runtime の backend auth は依然として PoC 状態にある。

- `backend/app/presentation/api/auth_api.py` は `/auth/*` のみを公開している
- request / response は `login_id`、`access_token`、`is_authenticated` 前提のままである
- `backend/app/application/use_cases/auth_usecase.py` は `admin/pass` をハードコードしている
- `backend/app/infrastructure/security/security_service_impl.py` は JWT cookie を読む旧 helper を持っている
- `GET /users/me` と `DELETE /users/me` は未実装である
- `backend/app/main.py` は `/api/v1` 配下に router を載せていない

この状態では、frontend が期待する正式 auth contract と backend runtime contract が食い違ったままで、ブラウザ上での login / status / logout / account deletion を確認できない。  
IMP-101 の役割は、PoC を増築せず正式仕様へ置換し、後続の IMP-102 / IMP-103 / IMP-104 が前提にできる backend 契約を確定させることにある。

### 完了条件

IMP-101 は、以下を満たした時点で完了とする。

1. `POST /api/v1/auth/signup` が email / password で user を作成し、`201` の正式 response を返すこと
2. `POST /api/v1/auth/login` が email / password を検証し、session を作成し、`asset_management_session` Cookie と `200` の正式 response を返すこと
3. `POST /api/v1/auth/logout` が current session のみ revoke し、Cookie を削除して `200` を返すこと
4. `GET /api/v1/auth/status` が通常未ログインを `200 authenticated=false`、期限切れのみを `401 SESSION_EXPIRED` で返すこと
5. `GET /api/v1/users/me` が current user の `id` と `email` を返すこと
6. `DELETE /api/v1/users/me` が logical delete、全 session revoke、Cookie 削除を行い、`200 deleted=true` を返すこと
7. `main.py` の router 構成が `/api/v1` 配下に揃っていること
8. backend tests が正式仕様ベースへ更新され、Docker 内で通ること
9. password reset、frontend、migration 追加を本タスクに含めないことが明記されていること
10. `login_id`、`access_token`、旧 `/auth/*` 契約の runtime 互換を残さないことが明記されていること

### 実装内容

#### Step 1: IMP-101 の対象外を最初に固定する

このタスクでは、以下を **やらない** と最初に固定する。

- `POST /api/v1/auth/password-reset/request`
- `POST /api/v1/auth/password-reset/reset`
- frontend の auth 画面、middleware、Redux の更新
- DB schema 変更や Alembic revision の追加
- PoC runtime 契約を残すための互換 endpoint 追加
- session 基盤そのものの再設計

IMP-101 は「PoC auth runtime を正式仕様へ置換する」ことに限定する。

#### Step 2: router 構成を `/api/v1` と resource 単位へ揃える

router は resource ごとに責務を分ける。

- `auth_api.py`
  - `POST /auth/signup`
  - `POST /auth/login`
  - `POST /auth/logout`
  - `GET /auth/status`
- `users_api.py`
  - `GET /users/me`
  - `DELETE /users/me`

`main.py` では `/api/v1` prefix の router を公開し、frontend の期待する base URL と揃える。  
旧 unversioned route を runtime 互換として残さない。

#### Step 3: request / response schema と DTO を正式仕様へ置換する

presentation schema と application DTO を、公開仕様どおりの shape に置き換える。

- signup request: `email`, `password`
- login request: `email`, `password`
- signup response:
  - `data.user_id`
  - `data.email`
  - `meta.message`
- login response:
  - `data.user.id`
  - `data.user.email`
  - `meta.message`
  - `meta.session_expires_in_days`
- logout response:
  - `data.logged_out`
  - `meta.message`
- status response:
  - `data.authenticated`
  - `data.user | null`
- users me response:
  - `data.id`
  - `data.email`
- delete me response:
  - `data.deleted`
  - `meta.message`

判断ルールは次のとおり。

- `access_token` は response body に含めない
- `is_authenticated` は top-level field に残さない
- `name` は auth / users me の公開 response 必須項目にはしない

#### Step 4: PoC `AuthUsecase` を正式 auth usecase へ置換する

`AuthUsecase` は IMP-003 の lower-level 基盤を使う正式実装へ置換する。

- signup
  - email uniqueness を確認する
  - password 要件を検証する
  - password hash を作成する
  - user を作成する
  - session は作成しない
- login
  - email で user を取得する
  - `deleted_at` が入っている user は認証失敗とする
  - password hash を検証する
  - raw session token を生成し hash を DB 保存する
  - session 有効期限は 30 日とする
  - response 用の user DTO と raw token を返す
- logout
  - current session token hash を使って current session のみ revoke する
  - session が見つからなくても Cookie 削除レスポンスは返せる構成にする
- auth status
  - Cookie 不在、token 不一致、revoke 済み、論理削除済み user は `authenticated=false`
  - session 期限切れのみ `SESSION_EXPIRED`
  - refresh 間隔を超えている場合だけ `touch` で延長する

#### Step 5: `users/me` と `delete me` 用 usecase を分離する

`GET /users/me` と `DELETE /users/me` は別 router かつ別 usecase で扱う。

- `UserAccountUsecase` を追加する
  - `get_me(current_user_id)`
  - `delete_me(current_user_id, current_session_token)`
- delete me の処理順は次で固定する
  1. current user を取得する
  2. `users.deleted_at = now` に更新する
  3. user に紐づく active session を全 revoke する
  4. response で auth cookie を削除する

email の再利用は許可しない。  
deleted user の login も許可しない。

#### Step 6: current user 解決を JWT helper から session ベースへ切り替える

`di/auth.py` の current user dependency は、旧 `get_current_user_from_cookie()` に依存しない構成へ置換する。

- request から `asset_management_session` Cookie を読む
- raw token を hash 化する
- active session を repository で引く
- 期限切れか通常未ログインかを判別できる結果を返す
- `users/me` と `delete me` の protected endpoint はこの dependency を正として使う

判断ルールは次のとおり。

- session refresh は current user dependency または status usecase のどちらか 1 か所に寄せ、二重更新しない
- expired session のみを通常未ログインと区別する
- current user 取得が必要な endpoint で Infrastructure を直接 import しない

#### Step 7: Cookie 制御を正式仕様へ揃える

login / logout / expired status / delete me で Cookie 制御を正式化する。

- Cookie 名は `asset_management_session`
- `HttpOnly`
- `SameSite=Lax`
- HTTPS 環境では `Secure=True`
- local HTTP 開発では `Secure=False`

logout と delete me と expired status では、client 側に cookie deletion が伝わるように同じ cookie 名で明示的に削除する。

#### Step 8: error handling を正式仕様へ揃える

少なくとも以下を正式化する。

- duplicate email: `409 DUPLICATE_NAME`
- invalid password input: `422 VALIDATION_ERROR`
- invalid credentials: `401 UNAUTHORIZED`
- expired session only: `401 SESSION_EXPIRED`
- unauthenticated normal case:
  - `GET /auth/status` は `200 authenticated=false`
  - protected `GET /users/me` と `DELETE /users/me` は `401 UNAUTHORIZED`

error body は `API_SPECIFICATION.md` の `error.code / error.message / error.details` 形式に揃える。

#### Step 9: Docker 前提で backend tests を正式仕様へ更新する

少なくとも以下を正式仕様ベースへ更新する。

- `backend/tests/application/test_auth_usecase.py`
- `backend/tests/presentation/test_auth_api.py`
- `backend/tests/presentation/test_users_api.py`

加えて、回帰確認として以下を流す。

- `backend/tests/infrastructure/test_user_repository_impl.py`
- `backend/tests/infrastructure/test_user_session_repository_impl.py`

### 実装時の判断ルール

本タスクでは、以下の判断を固定する。

- IMP-101 は backend runtime 契約の正式化を対象とする
- auth の正本は PoC ではなく `API_SPECIFICATION.md` と `AUTH_IMPLEMENTATION_DETAIL.md`
- signup は自動ログインしない
- login は複数端末ログインを許可する
- logout は current session のみ revoke する
- delete me は logical delete + 全 session revoke
- normal unauthenticated と expired session を区別する
- expired session だけ `401 SESSION_EXPIRED`
- protected endpoint の current user 解決は DI 経由にする
- frontend 互換のために旧 runtime route / field を残さない

### 非対象

以下は IMP-101 の対象外とする。

- `POST /api/v1/auth/password-reset/request`
- `POST /api/v1/auth/password-reset/reset`
- frontend の auth 画面実装
- frontend middleware や store の追加変更
- DB schema 変更と migration 追加
- email 送信基盤
- 旧 JWT 実装ファイルの全面削除だけを目的にした整理
